## 📋 建议的提示词补充内容

### **配置管理和方法保存原则:**

```markdown
## 配置保存和方法更新原则

当完成复杂的系统配置、部署或开发环境设置时，必须：

1. **立即保存成功配置**
   - 创建详细的配置文件备份
   - 记录所有关键参数和依赖
   - 包含完整的环境变量和端口配置
   - 保存所有自定义脚本和配置文件

2. **创建可重复的部署方法**
   - 编写一键部署脚本 (如 quick-deploy.sh)
   - 包含完整的前置条件检查
   - 自动化验证部署成功的测试
   - 提供清晰的错误恢复方案

3. **文档化最佳实践**
   - 创建 DEPLOYMENT-GUIDE.md 类型的详细指南
   - 记录 CONFIG-SNAPSHOT.md 类型的配置快照
   - 编写 LESSON-LEARNED.md 记录经验教训
   - 建立故障排除知识库

4. **持续更新维护**
   - 任何方法变更时立即更新相关文档
   - 新发现的问题要补充到故障排除指南
   - 定期验证保存的配置和脚本的有效性
   - 保持部署方法与技术栈变化同步

5. **投资回报最大化**
   - 优先保存复杂度高、重复性强的配置
   - 创建标准化模板供未来项目使用
   - 将成功经验抽象为可复用的方法论
   - 建立团队共享的配置知识库
```

---

## 🎯 具体实施建议

### **在系统提示词中加入触发条件:**

```markdown
当用户请求或你完成以下类型的任务时，自动执行配置保存流程：

- 复杂的多服务部署配置
- 开发环境的初始化设置  
- 数据库、容器、微服务的配置
- 工作流引擎(如n8n)的setup
- CI/CD流水线的建立
- 复杂的API集成配置
- 跨系统的认证和权限设置

触发后必须：
1. 创建 DEPLOYMENT-GUIDE.md
2. 创建 CONFIG-SNAPSHOT.md  
3. 创建 quick-deploy.sh 或类似自动化脚本
4. 创建验证测试脚本
5. 创建 LESSON-LEARNED.md
```

## 🔄 方法更新的触发机制

### **变更检测和文档更新:**

```markdown
当发现以下情况时，必须更新相关配置文档：

变更类型 → 需要更新的文件
- 端口修改 → CONFIG-SNAPSHOT.md, quick-deploy.sh
- URL变更 → DEPLOYMENT-GUIDE.md, 配置文件
- 新增服务 → 所有相关文档和脚本
- 依赖更新 → requirements文件, 部署脚本
- 故障修复 → DEPLOYMENT-GUIDE.md故障排除章节
- 流程优化 → quick-deploy.sh, LESSON-LEARNED.md

更新检查清单：
□ 相关配置文件是否需要修改？
□ 部署脚本是否需要更新？  
□ 故障排除指南是否需要补充？
□ 快速部署流程是否仍然有效？
□ 文档版本号是否需要更新？
```

## 🎓 学习要点集成

### **加入系统知识库的原则:**

```markdown
每次配置保存都应该包含：

1. 环境信息 (OS, 版本, 依赖)
2. 端口分配和网络配置
3. 数据库连接字符串
4. 认证凭据配置方法
5. 服务启动顺序和依赖关系
6. 验证成功的具体标准
7. 常见错误和解决方案
8. 性能调优参数

标准化文件结构：
/project-root/
├── DEPLOYMENT-GUIDE.md      # 详细部署指南
├── CONFIG-SNAPSHOT.md       # 配置文件备份
├── quick-deploy.sh          # 一键部署脚本
├── stop-services.sh         # 服务停止脚本
├── test-deployment.py       # 部署验证测试
├── LESSON-LEARNED.md        # 经验教训总结
└── troubleshooting/         # 故障排除知识库
```
